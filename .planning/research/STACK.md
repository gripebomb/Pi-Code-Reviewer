# Stack Research

**Domain:** Pi-installable code reviewer skill/package for repository audits
**Researched:** 2026-04-19
**Confidence:** HIGH

## Recommended Stack

### Core Technologies

| Technology | Version | Purpose | Why Recommended |
| ---------- | ------- | ------- | --------------- |
| Node.js | 20+ LTS | Package runtime for helper scripts and local development | Pi packages installed from npm/git run through npm workflows, and helper scripts are easiest to ship and test in the Node ecosystem. |
| TypeScript | 5.x | Author any helper scripts, validation utilities, and optional future extension code | Pi extensions are TypeScript-first, and TS keeps package logic maintainable even if v1 is mostly skill-driven. |
| Markdown skill (`SKILL.md`) following Agent Skills frontmatter | Agent Skills standard current in Pi docs | Primary user-facing workflow invoked with `/skill:name` | The user explicitly wants `/skill`, and Pi natively discovers skills from package `skills/` directories or `pi.skills` manifest entries. |
| npm package with `pi` manifest and `pi-package` keyword | npm package.json metadata | Distribution through npm and `pi install` | Official Pi package docs recommend a `pi` key in `package.json` or conventional directories, plus `pi-package` for discoverability in the gallery. |

### Supporting Libraries

| Library | Version | Purpose | When to Use |
| ------- | ------- | ------- | ----------- |
| `tsx` | 4.x | Run TypeScript helper scripts without a build step during development | Use in local development and tests if helper scripts are authored in TypeScript. |
| `vitest` | 2.x | Test helper scripts, report generation, and fixture-based review workflows | Use for package-level regression tests across sample repos and output formats. |
| `fast-glob` | 3.x | File discovery in optional helper scripts | Use only if the skill outgrows plain `find`/`rg` shell commands and needs deterministic cross-platform scanning logic. |
| `zod` | 3.x | Validate structured intermediate data if a helper script produces normalized findings JSON before markdown rendering | Use only if v1 introduces a scriptable finding pipeline instead of pure model-authored markdown. |

### Development Tools

| Tool | Purpose | Notes |
| ---- | ------- | ----- |
| npm | Publish, install, and pack the package | Needed for both direct npm install and `pi install npm:...` flows. |
| `npm pack` | Verify the tarball actually contains skills, references, scripts, and assets | Critical before publish; Pi can only load what the tarball includes. |
| `git` | Enable changed-files-focused review mode | v1 should use `git diff`, `git status`, and related read-only commands when available. |
| `rg` / `find` | Fast repo inspection from within the skill | Prefer these over bespoke scanning code unless portability problems force helper scripts. |

## Installation

```bash
# Package development
npm init -y
npm install -D typescript tsx vitest @types/node

# Optional supporting runtime libraries (only if needed)
npm install fast-glob zod
```

### Distribution shape

```bash
# Install globally via npm (package available on PATH / registry)
npm install -g pi-code-reviewer

# Install into Pi as a package
pi install npm:pi-code-reviewer

# Project-local Pi install
pi install -l npm:pi-code-reviewer
```

## Alternatives Considered

| Recommended | Alternative | When to Use Alternative |
| ----------- | ----------- | ----------------------- |
| Skill-first package | Extension-first package | Use an extension only if v1 truly needs custom tools, custom UI, or session-persistent state beyond what a skill plus built-in tools can handle. |
| Conventional `skills/` directory plus `pi.skills` manifest | Manifest-only packaging | Fine if the package layout is unusual, but conventional directories are simpler and easier to inspect. |
| Built-in Pi tools (`read`, `write`, `bash`, `grep`, `find`, `ls`) with light helper scripts | Heavy custom parser/analyzer runtime | Use a heavier runtime only after v1 proves the built-in toolchain cannot produce stable, actionable output. |
| Markdown outputs generated by the skill | Structured JSON-first pipeline | Add JSON later if machine consumption becomes a strong requirement; the user requested markdown checklist output for v1. |

## What NOT to Use

| Avoid | Why | Use Instead |
| ----- | --- | ----------- |
| Extension-first architecture in v1 | Adds code surface, runtime complexity, and more packaging/test burden before proving the core review workflow | Start with a skill package and only add an extension when there is a concrete gap. |
| `devDependencies` for runtime-critical code | Pi installs packages with production-oriented dependency handling; runtime code must be available after install | Put runtime dependencies in `dependencies`. |
| Skill names or directory names that do not match Agent Skills rules | Pi warns on invalid names and may fail to discover or load skills as expected | Use a lowercase, hyphenated directory/name pair such as `code-reviewer`. |
| Depending on agent auto-invocation for core usability | Pi can auto-load skills, but docs explicitly note models do not always do this reliably | Make `/skill:code-reviewer` the canonical invocation path. |

## Stack Patterns by Variant

**If v1 stays skill-only (recommended):**
- Use a package with `skills/code-reviewer/SKILL.md`, reference docs, and optional shell/Node helper scripts.
- Because Pi already exposes the built-in tools needed for repo inspection and file generation.

**If v1 later needs stronger determinism:**
- Add a small extension that registers a dedicated review tool or command while keeping the skill as the main UX entry point.
- Because extensions are the right fit for custom tools, UI, and persistent state, but they are not required to ship the first version.
