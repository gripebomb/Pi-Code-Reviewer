  <action>
Create `skills/code-reviewer/references/evidence-gathering.md` that defines the evidence gathering algorithm for Phase 2 of the skill workflow.

Content must include:
1. **File Read Order**: Specify breadth-first exploration sequence:
   - Start with root directory listing and key config files (package.json, tsconfig.json, pyproject.toml, Makefile, etc.)
   - Read entry points (main, index, app, server files)
   - Identify test directories and read sample test files
   - Read README and documentation files
   - Identify technology stack and conventions
   
2. **Depth-First Triggers**: Specify when to go deeper:
   - Files with complex logic (>50 lines, >3 nesting levels)
   - Files with security-sensitive operations (auth, input handling, file access)
   - Files with error handling paths
   - Central modules imported by many others
   - Files with TODO/FIXME comments
   
3. **Evidence Types to Collect**:
   - Structural: file organization, module boundaries, import patterns
   - Behavioral: error handling, edge cases, state management
   - Qualitative: naming quality, comment usefulness, complexity signals
   - Configurational: security settings, build configs, dependency versions
   - Test-related: test existence, coverage patterns, assertion quality
   
4. **Sufficiency Criteria**: When to stop gathering:
   - All entry points read
   - At least one test file examined (if tests exist)
   - Security-sensitive files identified and read
   - Key config files read
   - Evidence map covers all 5 categories
   
5. **Evidence Map Format**: How to organize collected evidence:
   - Category-tagged observations
   - File-path-indexed notes
   - Severity signals (potential High/Medium/Low indicators)
   - Cross-references between related observations

The guide should be concrete and actionable — an executing agent should know exactly what to read and in what order. Reference `references/scope-detection.md` for respecting `candidate_paths`.

  <action>
Create `skills/code-reviewer/references/category-analysis-guide.md` that operationalizes the review rubric into actionable inspection steps.

For EACH of the 5 categories, include:

**Code Quality**:
- Inspection steps: check error handling (unhandled promises, missing try/catch), complexity (nesting depth, function length), consistency (naming, async patterns), dead code (unused imports, unreachable branches), type safety
- Concrete signals: specific code patterns to look for
- Evidence format: what to record (line numbers, function names, error paths)
- Uncertainty handling: when language features make assessment unclear

**Refactoring**:
- Inspection steps: scan for duplication (>3 similar lines), coupling (circular imports, global state), abstraction level (mixed responsibilities), module boundaries, data flow clarity
- Concrete signals: specific structural patterns that indicate refactoring need
- Evidence format: file paths of duplicates, coupling metrics
- When NOT to refactor: simple designs that don't need abstraction

**Documentation**:
- Inspection steps: README completeness (install, run, usage), API docs, inline comments on non-obvious logic, architecture guidance, examples
- Concrete signals: missing sections, stale comments, undocumented public APIs
- Evidence format: specific missing sections, line references for stale comments
- Uncertainty handling: when conventions are obvious to domain experts

**Security**:
- Inspection steps: input validation, secrets handling (hardcoded keys, tokens), auth checks, dependency risk, error exposure, configuration safety
- Concrete signals: specific dangerous patterns (eval, exec, raw SQL, etc.)
- Evidence format: exact lines with issues, dependency versions
- Uncertainty handling: when security controls are implied but not visible

**Test Coverage**:
- Inspection steps: test existence, framework identification, coverage breadth, behavior verification, edge cases, CI integration
- Concrete signals: untested critical paths, brittle tests, missing error-path tests
- Evidence format: test file paths, uncovered function names
- Uncertainty handling: when tests might exist outside reviewed scope

Each category section should:
- Reference `references/review-rubric.md` for criteria
- Provide 3-5 concrete inspection steps
- List specific signals/patterns to look for
- Define evidence recording format
- Include uncertainty handling guidance

Cross-reference `references/evidence-gathering.md` for the evidence collection process.

  <action>
Create `skills/code-reviewer/references/evidence-synthesis.md` and update `skills/code-reviewer/SKILL.md` Phase 2.

**Evidence Synthesis Rules** (evidence-synthesis.md):
1. **Concrete over Vague**: Every finding must tie to at least one specific file/path/line. No findings without evidence.
2. **Pattern over Instance**: Single occurrence = Low severity; repeated pattern (>2 files) = Medium/High.
3. **Impact over Count**: One critical security issue outweighs ten style issues.
4. **Uncertainty Explicit**: When evidence is insufficient, state: "Evidence is limited: [specific gap]"
5. **Contextual Severity**: Same pattern can be High in security-critical code, Low in utilities.
6. **Cross-Category Consolidation**: Related findings across categories should reference each other.
7. **Finding Format**: Each finding must have: bold title, file/path reference, multi-line description with impact.

Include examples of good vs. bad findings for each rule.

**Update SKILL.md Phase 2**:
Replace the current Phase 2 "Gather Evidence" section with specific instructions:
- Add: "Before gathering evidence, read references/evidence-gathering.md for the exact file read order, depth triggers, and sufficiency criteria."
- Add: "Build an evidence map organized by category with file-path-indexed observations."
- Add: "Tag each observation with a preliminary severity signal (potential High/Medium/Low)."
- Add: "Prefer breadth first: read representative files across all categories before deep-diving into any single file."
- Add: "When evidence is limited for a category, record the evidence gap explicitly rather than forcing findings."
- Keep the existing status message: "◆ Gathering evidence from repository..."

Ensure the updated Phase 2 references the new evidence-gathering.md document.
